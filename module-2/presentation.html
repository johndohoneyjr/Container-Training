Docker Desktop for Windows

WSL2 Integration

Prerequisites:

  • Windows 10/11 Pro, Enterprise, or Education (64-bit)
  • Build 19044 or higher
  • Hardware virtualization enabled in BIOS
  • Minimum 4GB RAM (8GB+ recommended)

Quick Setup Commands:

# Enable WSL2 (PowerShell as Administrator)
wsl --install
wsl --set-default-version 2

# Verify installation
wsl --list --verbose
docker --version
docker run hello-world

Docker Desktop for macOS

Intel vs Apple Silicon

Alpine vs. Debian Images

Choosing the Right Base Image

Aspect Alpine Debian (Bookworm)
Size Smaller (~115MB runtime) Larger (~210MB runtime)
C Library musl libc glibc
Compatibility May have issues with native deps Better compatibility
Security Smaller attack surface Larger but well-maintained
Best For Simple .NET apps, size matters Complex deps, production

Recommendation: Start with Debian for compatibility, switch to Alpine if size is critical and tested.

Docker Hardened Images

Enhanced Security for Production

What are Docker Hardened Images?

Hardened images from dhia.io provide enhanced security:

  • Pre-hardened base images: Security configurations applied
  • Non-root by default: Runs as unprivileged user
  • Minimal attack surface: Unnecessary packages removed
  • Regular security updates: Actively maintained
  • Compliance ready: Meets security standards

Example Usage:

# Instead of:
FROM mcr.microsoft.com/dotnet/aspnet:10.0

# Use hardened:
FROM dhia.io/dotnet/aspnet:10.0-alpine

Multi-Stage Dockerfiles

Optimize Image Size

Why Multi-Stage?

  • Smaller final image: Only runtime needed, not SDK
  • Better security: No build tools in production
  • Faster deployment: Smaller images transfer faster
  • Clear separation: Build vs. runtime concerns

Basic Pattern:

# Build stage - SDK image
FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
WORKDIR /src
COPY *.csproj .
RUN dotnet restore
COPY . .
RUN dotnet publish -c Release -o /app

# Runtime stage - Smaller runtime image
FROM mcr.microsoft.com/dotnet/aspnet:10.0
WORKDIR /app
COPY --from=build /app .
ENTRYPOINT ["dotnet", "MyApp.dll"]

Optimization Techniques

1. Layer Caching Optimization:

2. Use .dockerignore:

2.4 Building and Running Containers

Docker Build Commands

Basic Build:

# Build with tag
docker build -t myapp:1.0 .

# Build with specific Dockerfile
docker build -f Dockerfile.prod -t myapp:1.0 .

# Build with build arguments
docker build --build-arg VERSION=1.0 -t myapp:1.0 .

# Build for multiple platforms
docker buildx build --platform linux/amd64,linux/arm64 -t myapp:1.0 .

Important Options:

  • -t: Tag the image
  • -f: Specify Dockerfile
  • --no-cache: Build without cache
  • --target: Build specific stage

Running .NET Containers

Docker Run Commands

Common Run Options:

# Run in detached mode with port mapping
docker run -d -p 8080:8080 --name myapi myapp:1.0

# Run with environment variables
docker run -d -p 8080:8080 \
  -e ASPNETCORE_ENVIRONMENT=Production \
  -e ConnectionStrings__Default="Server=db;..." \
  --name myapi myapp:1.0

# Run with volume mount
docker run -d -p 8080:8080 \
  -v $(pwd)/data:/app/data \
  --name myapi myapp:1.0

# Run with resource limits
docker run -d -p 8080:8080 \
  --memory="512m" --cpus="1.0" \
  --name myapi myapp:1.0

Container Lifecycle Management

Essential Commands

# View running containers
docker ps

# View all containers (including stopped)
docker ps -a

# View container logs
docker logs myapi
docker logs -f myapi  # Follow logs

# Execute command in running container
docker exec -it myapi /bin/sh
docker exec myapi dotnet --info

# Stop container
docker stop myapi

# Start stopped container
docker start myapi

# Restart container
docker restart myapi

# Remove container
docker rm myapi
docker rm -f myapi  # Force remove running container

    ports:
      - "8080:8080"

2.5 .NET Container Best Practices

Security Hardening

Essential Security Practices:

  • Use official images: MCR or Docker Hardened Images only
  • Specific version tags: Avoid latest in production
  • Non-root user: Run as unprivileged user
  • Multi-stage builds: Remove build tools from final image
  • Scan for vulnerabilities: Docker Scout, Trivy, Snyk
  • Health checks: Implement proper health endpoints
  • Secret management: Never hardcode secrets
  • Read-only filesystem: When possible
  • Drop capabilities: Minimal Linux capabilities

Scanning for Vulnerabilities

Security Scanning Tools

Docker Scout (Built-in):

# Scan image for vulnerabilities
docker scout cves myapp:1.0

# View recommendations
docker scout recommendations myapp:1.0

# Compare images
docker scout compare --to myapp:2.0 myapp:1.0

Trivy (Open Source):

# Install Trivy
brew install aquasecurity/trivy/trivy  # macOS
winget install Aqua.Trivy              # Windows

# Scan image
trivy image myapp:1.0

# Scan with severity filter
trivy image --severity HIGH,CRITICAL myapp:1.0

Generate SBOM:

docker sbom myapp:1.0 > sbom.json

Performance Optimization

Making .NET Containers Faster

Build-Time Optimizations:

  • Layer caching: Order COPY/RUN efficiently
  • .dockerignore: Exclude unnecessary files
  • Alpine images: Smaller = faster transfer
  • Parallel builds: Use BuildKit

.NET Specific Optimizations:

  • ReadyToRun (R2R): Pre-compiled for faster startup 
    dotnet publish -c Release -r linux-x64 -p:PublishReadyToRun=true
  • Trimming: Remove unused assemblies 
    dotnet publish -c Release -p:PublishTrimmed=true
  • AOT Compilation: Native compilation (.NET 8+)
